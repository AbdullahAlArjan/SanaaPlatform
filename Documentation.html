Sanaa Platform — Developer Documentation

صنعة is a full-stack freelance service marketplace connecting clients with skilled freelancers. Clients browse and order services, freelancers manage their profiles and incoming work, and admins oversee the entire platform.

1 Project Overview

Sanaa is a two-sided marketplace where:

  • Clients browse a service catalog, place orders, save favorites, and leave reviews.
  • Freelancers create profiles, list services, manage incoming orders, and upload portfolio images.
  • Admins approve freelancer profiles, manage users, view platform-wide stats, and resolve reports.

3 Architecture Overview

The backend follows a strict 3-Tier Architecture. Each layer has a single responsibility and communicates only with its adjacent layer.

┌─────────────────────────────────────────────────────┐ │ CLIENT (Browser) │ │ HTML + CSS + Vanilla JS (ES Modules) │ └───────────────────────┬─────────────────────────────┘ │ HTTP (JSON + JWT) ▼ ┌─────────────────────────────────────────────────────┐ │ Sanaa.API (Presentation Layer) │ │ Controllers · Program.cs · DTOs · SignalR Hubs │ │ • Routes HTTP requests & enforces [Authorize] │ │ • Validates input via DTOs │ │ • Calls BLL services via injected interfaces │ └───────────────────────┬─────────────────────────────┘ │ Interface contracts (DI) ▼ ┌─────────────────────────────────────────────────────┐ │ Sanaa.BLL (Business Logic Layer) │ │ Interfaces · Services · DTOs │ │ • Implements all business rules │ │ • Orchestrates calls to DAL │ │ • Maps Entities ↔ DTOs · Sends emails & alerts │ └───────────────────────┬─────────────────────────────┘ │ EF Core DbContext ▼ ┌─────────────────────────────────────────────────────┐ │ Sanaa.DAL (Data Access Layer) │ │ SanaaDbContext · Entities · Migrations │ │ • Defines DB schema via Code-First │ │ • Fluent API configuration & migrations │ └───────────────────────┬─────────────────────────────┘ │ SQL ▼ ┌──────────────┐ │ SanaaDB │ │ SQL Server │ └──────────────┘
Dependency Direction Rule: API → BLL → DAL — a lower layer never imports from a higher one. The API project only knows about BLL interfaces; it never touches SanaaDbContext directly.

Dependency Injection

All services are registered in Program.cs as Scoped (one instance per HTTP request):

csharp · Program.cs
builder.Services.AddScoped<IOrderService,     OrderService>();
builder.Services.AddScoped<IFavoritesService, FavoritesService>();
builder.Services.AddScoped<IFreelancerService, FreelancerService>();
// ... all 15 interfaces registered the same way

7 Frontend Architecture

The entire frontend is coordinated by two central files that every protected page imports:

js/api.js — The HTTP Client

All API calls go through apiFetch(). It automatically:

  • Prepends the base URL (https://localhost:7101)
  • Injects the Authorization: Bearer <token> header
  • Handles 401 responses by silently refreshing the access token and retrying (with a queue to prevent race conditions)
  • Redirects to Login.html if refresh also fails
javascript · api.js usage
import { apiFetch } from './api.js';

// GET — auto auth header injected
const orders = await apiFetch('/api/Orders/my-orders?pageNumber=1&pageSize=10');

// POST with body
await apiFetch(`/api/Favorites/${serviceId}`, { method: 'POST' });

// File upload
await apiUpload('/api/Freelancers/profile-image', formData);

js/auth.js — Auth State & Route Guards

FunctionDescription
login(email, password)Calls /api/auth/login, stores token + user object in localStorage
logout()Calls /api/auth/logout, clears localStorage
requireAuth(allowedRoles)Route guard — returns user object or redirects to login
getCurrentUser()Returns parsed user object from localStorage
redirectByRole(role)Navigates to the correct dashboard by role
javascript · requireAuth pattern
import { requireAuth } from './auth.js';
import { apiFetch }    from './api.js';

document.addEventListener('DOMContentLoaded', async () => {
    // Not logged in → auto redirect to Login.html
    // Wrong role   → auto redirect to correct dashboard
    const user = requireAuth(['client']);
    if (!user) return; // halt execution

    // ✅ Safe to proceed — fully authenticated
    const favorites = await apiFetch('/api/Favorites');
    renderFavorites(favorites);
});

Adding a Favorites Heart Button

javascript · addToFavorites
// post.js — window.* so inline onclick can reach it despite module scope
window.addToFavorites = async function(serviceId, btnElement) {
    const user = requireAuth(['client']);
    if (!user) return;

    await apiFetch(`/api/Favorites/${serviceId}`, { method: 'POST' });

    // Visual feedback: outline → solid red heart
    btnElement.querySelector('i').classList.replace('far', 'fas');
    btnElement.querySelector('i').style.color = '#e74c3c';
};

8 Authentication Flow

Registration & Email Verification

flow
1. POST /api/auth/register   → account created (IsEmailVerified = false)
2. POST /api/auth/send-otp   → OTP emailed to user
3. POST /api/auth/verify-otp → IsEmailVerified = true
4. Redirect to Login page

Login & Token Lifecycle

flow
1. POST /api/auth/login
   ← { accessToken, refreshToken, user }

2. Store both tokens in localStorage

3. Every API request:
   → Authorization: Bearer <accessToken>

4. On 401 (token expired):
   → api.js queues the failing request
   → POST /api/auth/refresh  { refreshToken }
   ← { accessToken, refreshToken (rotated) }
   → Update localStorage
   → Retry all queued requests

5. On refresh failure:
   → Clear localStorage
   → Redirect to Login.html

Password Reset

flow
1. POST /api/auth/forgot-password  { email }
2. User receives OTP via email
3. POST /api/auth/verify-otp       { email, code, purpose: "PasswordReset" }
4. POST /api/auth/reset-password   { email, code, newPassword }

9 Getting Started

Prerequisites

  • .NET 8 SDK
  • SQL Server (Express or Developer edition)
  • Visual Studio 2022 or VS Code
  • A modern browser (Chrome, Edge, Firefox)

Backend Setup

1
Open the solution
bash
cd Backend
start SanaaPlatform.sln    # opens in Visual Studio
2
Configure the connection string

Open Sanaa.API/appsettings.json and update DefaultConnection:

json · appsettings.json
"ConnectionStrings": {
  "DefaultConnection": "Server=.;Database=SanaaDB;Trusted_Connection=True;TrustServerCertificate=True;"
}
3
Apply migrations

In Visual Studio → Package Manager Console (set Default Project to Sanaa.DAL):

Package Manager Console
Update-Database

Or via .NET CLI:

bash
dotnet ef database update --project Backend/Sanaa.DAL --startup-project Backend/Sanaa.API
4
Run the API
bash
cd Backend/Sanaa.API
dotnet run

API starts at https://localhost:7101 · Swagger UI at https://localhost:7101/swagger

Frontend Setup

The frontend is plain HTML/CSS/JS — no build step required.

⚠️ ES6 modules (import/export) require a proper HTTP server — they do not work via file:// protocol.
A
VS Code Live Server (recommended)
  1. Install the Live Server extension
  2. Right-click index.html → Open with Live Server
  3. Browser opens at http://127.0.0.1:5500
B
Node.js serve
bash
cd Frontend/san3a-edit
npx serve .

First Run Checklist

  • SQL Server is running
  • Update-Database completed without errors (verify SanaaDB exists in SSMS)
  • API starts and Swagger loads at https://localhost:7101/swagger
  • Frontend served over HTTP (not file://)
  • API_BASE_URL in js/api.js matches the running API port

Appendix — Adding a New Feature

Follow this checklist for every new feature to maintain the 3-tier pattern:

  1. Create Entity class in Sanaa.DAL/Entities/ DAL
  2. Add DbSet<T> to SanaaDbContext.cs DAL
  3. Configure relationships in OnModelCreating() via Fluent API DAL
  4. Run: Add-Migration <Name> → Update-Database DAL
  5. Add response DTO to Sanaa.BLL/DTOs/ BLL
  6. Define interface in Sanaa.BLL/Interfaces/IYourService.cs BLL
  7. Implement in Sanaa.BLL/Services/YourService.cs BLL
  8. Create Controller in Sanaa.API/Controllers/ API
  9. Register: builder.Services.AddScoped<IYourService, YourService>() in Program.cs API
  10. Import { apiFetch } from ./api.js Frontend
  11. Import { requireAuth } from ./auth.js and guard the page Frontend
  12. Call apiFetch('/api/YourEndpoint', { method: 'POST', body: ... }) Frontend
